- 文档路径建议：`docs/01_prd/v1_reference_notes.md`
- 文档状态：Draft v0.2
- 更新日期：2026-04-20
- 用途：将现有竞品信息与当前项目共识收口为 **AI 销售助手 V1** 的研究输入，服务于后续 `ai_sales_assistant_v1_prd.md`、`02_specs` 与第一批正式 task 的拆分。

## 1. 文档目的

本文档不是完整竞品报告，也不是技术实现 spec。

它主要回答下面几个问题：

1. 当前 V1 到底要参考哪些产品，为什么是它们。
2. 每家产品分别值得学习哪一层，不该学哪一层。
3. 从竞品中可以提炼出哪些 V1 设计原则。
4. 竞品对当前系统架构与部署形态有什么启发。
5. 哪些内容应进入后续 PRD，哪些内容应明确延后。

本文档默认基于当前项目共识：

- 当前主线：**AI 销售助手 V1**
- 当前 V1 核心闭环：
  1. 产品学习
  2. 获客分析
  3. 结构化输出 / 报告
- 当前明确不优先推进：
  - 联系方式抓取
  - 自动触达
  - 完整 CRM
  - 大而全企业销售平台能力
- 当前系统路线：
  - **本地服务器承载正式后端**
  - **手机端仅作控制入口**
  - **OpenClaw 作为 agent runtime / execution layer**
  - **架构按未来可迁云设计**

## 2. 当前 V1 的研究边界

### 2.1 当前要研究的，不是“完整销售平台”

本阶段不是为了设计一个探迹 / 销氪 / HubSpot 式的大型企业销售平台。

当前 V1 只需要先回答：

- 用户如何通过对话，让 AI 快速理解“我卖什么、卖给谁、为什么有人会买”。
- AI 如何基于这一理解，产出一份有实际价值的获客分析结果。
- 结果如何以简单、清晰、可复用的方式呈现给用户。
- 在上述过程中，哪些内容应沉淀为正式业务对象，而不是停留在聊天记录中。

### 2.2 当前研究对象要围绕四类问题展开

#### A. 产品学习
研究 AI 如何帮助用户完成产品知识梳理，而不是要求用户先准备好大量结构化资料。

#### B. 获客分析
研究 AI 如何从“产品理解”过渡到“目标客户、场景、行业、上下游、潜客方向”的分析。

#### C. 结果交付
研究如何把分析结果交付成用户愿意保存、复看、继续编辑的结构化内容。

#### D. 系统承载方式
研究成熟竞品是如何处理：

- 权威数据主存
- 多端访问
- 团队协作
- 研究过程与业务主档的关系

## 3. 为什么当前优先参考这五家

当前优先参考：

1. Clay
2. Apollo.io
3. 纷享销客 SFA AI
4. 探迹科技
5. 销氪 CRM

选择标准不是“谁最强”，而是：

- 是否直接影响 V1 的核心能力设计
- 是否能提供某一层的最佳参考
- 是否能帮助我们判断什么该做、什么不该做
- 是否能对系统部署与数据承载方式形成启发

### 3.1 这五家的分工

| 产品 | 当前最值得参考的层 | 不应该照搬的层 |
|---|---|---|
| Clay | AI 研究代理、数据丰富化、工作流组织 | 积分制复杂计费、过重工作流复杂度 |
| Apollo.io | SMB 入口、免费增值、数据库+动作一体化包装 | 后续触达 / 序列能力的大量铺开 |
| 纷享销客 SFA AI | Agent 分层思路、对话式工作方式、统一后端可多部署 | 企业级复杂流程、行业模板深度定制 |
| 探迹科技 | 产业链 / 上下游分析、企业知识图谱视角 | 大规模企业数据基础设施 |
| 销氪 CRM | 多维筛选、线索评分、商机推送逻辑、平台化数据开放 | 大型 CRM 套件化能力 |

## 4. 五家重点竞品拆解

## 4.1 Clay

### 当前最值得学习的点

#### 1. “研究代理”而不是“静态查询工具”
Clay 值得参考的核心不是“有很多数据源”，而是它把研究过程产品化了。

对我们 V1 的启发：

- 获客分析不应只是一个搜索框 + 一堆原始结果。
- AI 应该承担“研究员”的角色，主动组织信息、提出假设、补充上下文、形成结论。
- 用户看到的应该是“分析结果”，而不只是“查到的信息”。

#### 2. 多数据源 / 多信号输入的架构思维
虽然 V1 不需要一开始就做真正的瀑布流丰富化，但 Clay 的思路提醒我们：

- 同一个潜在客户方向可以由多个信息源共同支撑。
- 后续系统应允许逐步扩展数据来源，而不是把数据结构写死。
- 分析结论要能显示“为什么得出这个判断”。

#### 3. 工作流式组织
Clay 的价值还在于它不是单点功能，而是把多个动作串起来。

对 V1 的启发：

- 产品学习 → 客群分析 → 输出报告，本质上也是一个链式工作流。
- V1 应把这个链路做得非常清楚，让用户感觉“我在完成一次完整分析”，而不是在点很多零碎功能。

### 当前不建议学习的点

#### 1. 积分制计费
现阶段不建议采用复杂 credits 体系。

理由：

- 用户成本感知差
- 价格预期不稳定
- 会增加理解门槛

#### 2. 复杂自动化工作流
Clay 的复杂度很高，不适合作为 V1 的交互基准。

V1 更应该是：

- 用户说清楚产品
- AI 自动组织分析
- 用户得到一份结构化结论

而不是让用户自己搭很多条件、节点、动作。

### 对 V1 的直接输入

- 需要一个“研究型分析 Agent”而不是简单问答机器人。
- 结果页要体现推理过程和依据，而不是只给答案。
- 后续数据层设计要预留多来源扩展能力。
- 研究过程与正式业务对象应分离。

## 4.2 Apollo.io

### 当前最值得学习的点

#### 1. SMB 友好入口
Apollo 非常值得学习的不是“数据量大”，而是它对中小团队更友好。

对 V1 的启发：

- 首次使用门槛必须低。
- 用户不应被大量企业配置流程拦住。
- 要尽量实现“打开即能开始描述产品、快速得到第一份结果”。

#### 2. 能力打包方式清晰
Apollo 的产品包装通常比较直接，用户容易理解“这个产品帮我干什么”。

对 V1 的启发：

- 首页与 onboarding 不能讲太多概念。
- 要明确告诉用户：你先告诉我产品，我帮你分析谁更可能成为客户。
- 不要用过多平台型、企业型术语压住用户。

#### 3. 平台内对象与平台外导出的区分
Apollo 给人的启发是：

- 平台内研究结果和平台外导出的正式数据，是两个层次。
- 数据使用、导出、同步、分享应该有边界。

### 当前不建议学习的点

#### 1. 大量触达动作并入 V1
Apollo 的数据库、序列、拨号等是一体化包装，但这些不应该成为当前 V1 的范围。

#### 2. 过度围绕“联系人数据库”设计体验
我们当前的主线不是先卖数据库，而是先卖“产品学习 + 获客分析能力”。

### 对 V1 的直接输入

- 首页和 onboarding 要极简。
- 第一轮分析必须快出结果。
- 平台内业务对象与未来导出 / 同步边界要分开。
- 产品包装要让个人 / SMB 用户看得懂。

---

## 4.3 纷享销客 SFA AI

### 当前最值得学习的点

#### 1. Agent 分层思路
纷享销客最值得参考的是“助理 / 顾问 / 总监”这种分层结构。

对 V1 的启发：

可以把当前 V1 抽成 3 个核心 Agent 角色：

1. **产品学习 Agent**
   - 负责理解产品、行业、优势、目标人群、典型使用场景
2. **获客分析 Agent**
   - 负责分析潜在客户方向、细分行业、上下游与场景机会
3. **报告输出 Agent**
   - 负责把分析结果整理成结构化文档 / 页面

#### 2. “对话驱动工作”而不是“后台系统驱动工作”
这对我们尤其重要。

V1 的关键差异不是“更多后台功能”，而是：

- 用户通过说话 / 输入描述就能启动整个分析流程。
- 系统主动追问并补全关键信息。
- 用户不需要先学会复杂的后台配置结构。

#### 3. 云端 / 专属云 / 私有化的统一产品思路
纷享销客最值得借鉴的一点是：

- 可以有不同部署形态
- 但产品逻辑、对象模型和系统边界仍然统一

这对我们当前路线的启发是：

- 第一阶段可以先用本地服务器承载正式后端
- 但对象模型、API、agent contract 应按未来可迁云设计
- 不能把“当前本地部署”做成“永久本地耦合”

### 当前不建议学习的点

#### 1. 大型企业销售流程深耕
纷享销客本身更偏企业深流程，V1 不应提前引入大量企业级字段和流程节点。

#### 2. 行业模板过重
V1 可以借鉴模板思路，但不应该一开始就做过度依赖预设行业模板的系统。

### 对 V1 的直接输入

- V1 内部要按 Agent / 阶段思维设计，而不是做成一锅粥。
- 对话式交互应贯穿产品学习与分析过程。
- 每个阶段的输出对象要明确可存储、可复用。
- 产品逻辑与部署形态要解耦。

## 4.4 探迹科技

### 当前最值得学习的点

#### 1. 产业链 / 上下游视角
探迹最值得学的不是“企业数据多”，而是它对产品上下游关系的理解方式。

对 V1 的启发：

- 获客分析不能只停留在“谁和你一样行业”。
- 还要分析：
  - 谁是你的直接客户
  - 谁是你的间接客户
  - 谁在你的上游 / 下游链路中存在合作可能
  - 哪些场景用户本来没想到，但其实存在需求

这会直接增强 V1 的“发现新机会”能力。

#### 2. 结构化行业理解
探迹的启发是：

- 产品画像不能只写一句话简介。
- 需要能逐步形成“产品—行业—客户—场景—上下游”的关系网。

### 当前不建议学习的点

#### 1. 自建超大规模企业数据底座
这不适合 V1，也不适合当前阶段投入。

#### 2. 全流程销售平台化
探迹已经是大平台，V1 不能把自己的节奏绑到这个复杂度上。

### 对 V1 的直接输入

- 获客分析模块中应加入“上下游 / 延伸场景 / 邻近行业机会”视角。
- Product Profile 不应只是单一产品说明，而应逐步形成关系型画像。
- 后端数据对象要允许这类关系视角逐步增强，而不是被当前字段写死。

---

## 4.5 销氪 CRM

### 当前最值得学习的点

#### 1. 多维筛选思维
销氪的启发在于：潜客分析最终必须能落到“筛选逻辑”上。

即使 V1 暂时不提供大型数据库搜索，也要在结果结构中体现：

- 哪些行业优先
- 哪些企业特征优先
- 哪些角色优先
- 哪些线索说明更值得优先跟进

#### 2. 线索评分模型思维
V1 虽然不需要完整 CRM，但可以在分析层先做“优先级判断”。

例如：

- 高潜在匹配行业
- 高价值使用场景
- 更可能快速成交的客户类型
- 需要长期教育的客户类型

#### 3. 平台化数据开放思维
销氪更像是通过 API / 平台层开放数据能力，而不是让各种端和脚本直接碰数据库。

对 V1 的启发：

- 手机端应通过正式后端接口工作。
- OpenClaw runtime 也应通过产品层接口读写业务对象。
- 不要让前端、脚本、agent 直接把本地文件或数据库当作事实源。

### 当前不建议学习的点

#### 1. 把 V1 变成 CRM
销氪的 CRM 化能力不应提前进入 V1 范围。

#### 2. 一上来就做极多筛选维度
当前阶段更适合先把分析结构做对，而不是先把维度堆满。

### 对 V1 的直接输入

- 分析结果中要有优先级与分层。
- 不只是“列出潜客方向”，还要给出“先做哪类、为什么”。
- 平台内读写应优先通过产品服务边界，而不是数据库直连。

## 5. 从竞品中提炼出的 V1 设计原则

以下原则应视为后续 PRD 的直接输入。

### 原则 1：V1 只做“产品学习 + 获客分析 + 结构化输出”

当前不做：

- 联系方式抓取
- 自动触达
- 大型 CRM
- 重企业协同流程

### 原则 2：差异化核心不是数据库，而是“AI 对话式产品学习”

V1 的第一竞争力不是“我有多少企业数据”，而是：

- 用户几乎不用准备复杂资料
- 只通过对话就能让 AI 建立产品理解
- 这种理解会成为后续分析的基础

### 原则 3：用户得到的应是“分析结果”，不是“原始搜索结果”

V1 应把 AI 定位成研究助手 / 分析顾问，而不是搜索框。

### 原则 4：首次使用必须足够快

目标应是：

- 用户在很短时间内完成第一次产品学习
- 在第一次会话中就拿到第一版获客分析结论

### 原则 5：产品结构应按阶段 / Agent 组织

建议最小结构：

1. 产品学习 Agent
2. 获客分析 Agent
3. 报告输出 Agent

### 原则 6：Product Profile 必须结构化

产品学习的结果不能只是聊天记录。

它至少应沉淀出可复用的结构化对象，例如：

- 产品是什么
- 核心卖点
- 目标客户类型
- 典型使用场景
- 行业属性
- 上下游 / 邻近场景机会
- 购买动因 / 拒绝点

### 原则 7：获客分析要有“优先级”而不是只列清单

V1 不能只是给一堆可能客户方向。

还要告诉用户：

- 哪一类优先
- 为什么优先
- 哪一类更适合马上尝试
- 哪一类需要长期培育

### 原则 8：分析结果要适合沉淀为文档

结果不应只停留在聊天气泡里。

应该可以沉淀成：

- 可复看
- 可编辑
- 可迭代
- 可复制分享

的结构化报告。

### 原则 9：端侧轻、后端统一

手机端应尽量轻：

- 发起
- 查看
- 轻量编辑
- 确认
- 管理历史入口

而权威对象与执行编排应以后端为准。

### 原则 10：当前部署先本地服务器，但架构必须可迁云

第一阶段可用本地服务器承载正式后端，这是阶段性部署选择。

但产品逻辑、对象模型、API 边界与 agent contract 必须按未来迁云设计。

### 原则 11：OpenClaw 是 runtime，不是产品本体

OpenClaw 更适合承担：

- 模型调用
- 工具调用
- agent 执行
- 运行时编排

不应直接承担：

- 产品业务主数据库
- 产品对象模型定义中心
- 产品权限与协作中心

### 原则 12：价格与价值必须容易理解

现阶段更适合：

- 免费试用 / 免费首份分析
- 固定月费
- 避免复杂积分体系

## 6. 当前明确“不学什么”

为了避免后续被成熟竞品带偏，当前应明确不学这些：

### 6.1 不学“大而全企业平台”路线
不把探迹 / 销氪 / 迈富时式的全链条平台能力作为当前 V1 目标。

### 6.2 不学“先做完整 CRM”
不提前引入客户管理、销售流程、协同审批、复杂报表等能力。

### 6.3 不学“先做自动触达”
不在 V1 中优先做电话、企微、短信、邮件序列自动化。

### 6.4 不学“重工作流搭建器”
不把用户变成流程搭建者。

### 6.5 不学“过度依赖预设模板”
虽然模板是有价值的，但 V1 的核心应是 AI 主动理解用户，而不是要求用户先匹配固定模板。

### 6.6 不学“手机端主存”
不把手机 App 当作产品权威数据库。

### 6.7 不学“本地部署即本地耦合”
即使当前阶段正式后端先部署在本地服务器，也不把系统做成：

- 只有本地文件路径才能工作
- 只有本机脚本直连才能工作
- 只有某台机器上的私有状态才是事实源

## 7. 当前建议的 V1 最小对象模型

本节不是正式 spec，只是给 PRD 和后续 task 提供研究输入。

## 7.1 ProductProfile（产品画像）

建议至少包含：

- `id`
- 产品名称
- 产品一句话描述
- 产品类别
- 核心卖点
- 目标客户类型
- 主要使用场景
- 当前成交模式 / 交付方式
- 购买动因
- 常见顾虑 / 阻力
- 所在行业
- 可延伸行业 / 邻近场景
- 上下游关系线索
- 当前已知优势与短板
- `status`
- `version`
- `updated_at`

## 7.2 LeadAnalysisResult（获客分析结果）

建议至少包含：

- `id`
- `product_profile_id`
- 推荐优先行业列表
- 推荐优先客户类型列表
- 典型使用场景列表
- 上下游 / 邻近行业机会
- 优先级排序
- 每项推荐的理由
- 风险与限制说明
- 下一步建议动作
- `status`
- `version`

## 7.3 AnalysisReport（分析报告）

建议至少包含：

- `id`
- `product_profile_id`
- `lead_analysis_result_id`
- 产品理解摘要
- 目标客户画像摘要
- 获客方向建议
- 优先级排序
- 场景 / 行业机会说明

## 8. 对当前系统架构与部署形态的启发

### 8.1 主流竞品几乎都不是手机端主存

从当前重点参考对象看，成熟销售智能 / CRM / AI 销售产品普遍是：

- 统一后端承载权威数据
- 多端访问统一对象
- 手机 App 作为入口或移动工作台
- 研究过程与正式主档分层

这意味着：

- 当前项目不应把手机端当作主数据库。
- 聊天过程不应直接等于业务真相。
- 正式对象应存放在统一后端。

### 8.2 当前路线与竞品启发是一致的

结合竞品研究与当前项目阶段，现阶段最合理的路线是：

- **本地服务器承载正式后端**
- **手机端仅作控制入口**
- **OpenClaw 作为 agent runtime / execution layer**
- **架构按未来可迁云设计**

这条路线的含义是：

- 现在先用本地服务器满足控制感、快速迭代和成本可控。
- 但系统对象模型、API 边界和 agent contract 不依赖本地部署本身。
- 后续如果需要云部署、专属云或正式 SaaS 化，不需要推翻产品逻辑重来。

### 8.3 当前最应该坚持的边界

- OpenClaw 负责运行与执行，不直接等于产品层。
- 产品层负责业务对象、状态、历史、结果沉淀。
- 手机端负责发起、查看、轻量编辑与确认。
- 权威版本以后端为准。

## 9. 给后续 PRD 的直接输入

后续 `ai_sales_assistant_v1_prd.md` 建议重点回答以下问题：

### 9.1 用户是谁

优先考虑：

- 中小企业老板 / 创始人
- 销售负责人
- 个人销售 / 商务拓展
- 还没有成熟销售体系，但需要更快找到潜在客户的人

### 9.2 用户第一条主路径是什么

建议主路径：

1. 用户在手机端描述自己的产品
2. AI 主动追问并补全产品理解
3. 后端形成 / 更新 ProductProfile
4. 用户确认 ProductProfile
5. 后端基于 ProductProfile 生成获客分析
6. 系统输出结构化分析报告
7. 用户在手机端查看、继续编辑或重新发起分析

### 9.3 第一版需要什么核心页面 / 结构

建议优先：

- 首页 / 开始分析入口
- 产品学习对话页
- 产品画像确认页
- 获客分析结果页
- 分析报告页
- 历史任务 / 历史报告入口（可轻量）

### 9.4 第一版明确不做什么

仍然明确不做：

- 联系方式抓取
- 自动触达
- 完整 CRM
- 复杂协作流
- 云端 / 本地双形态并行产品
- 手机端主存

### 9.5 第一版需要明确的架构前提

PRD 中应正式写清：

- 手机端是控制入口，不是主真相层。
- 正式后端由本地服务器承载。
- OpenClaw 是 runtime。
- 产品层通过正式对象和接口沉淀业务结果。
- 后续可迁云，但当前不以迁云本身为阶段目标。

## 10. 一句话总结

当前这份研究笔记的核心结论不是“我们应该复制哪家竞品”，而是：

> **AI 销售助手 V1 应以“AI 对话式产品学习”作为差异化核心，以“获客分析 + 结构化报告”作为最小闭环，并采用“本地服务器承载正式后端 + 手机端仅作控制入口 + 架构按未来可迁云设计”的系统路线。**
