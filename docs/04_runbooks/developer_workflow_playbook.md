# 开发者工作流手册
更新时间：2026-04-20
## 1. 文档定位
本文档是给**开发者本人**看的工作流手册，用于规范当前项目的推进方式。 
本文档回答的问题包括： - 我每天推进项目时，应该按什么顺序做 - 
产品方向变化时，应该先改什么 - 新功能开发时，应该先写什么文档 - 
功能完成后，如果还要改细节，应该怎么处理 - 什么时候让 Codex 
开工，什么时候不该直接让它改代码 本文档不是： - 产品 PRD - 仓库级 
agent 规则文件（那是 `AGENTS.md`） - 技术方案文档 - 单个任务文档 
它只负责说明：
> **开发者如何使用当前仓库、文档和 Codex，稳定地推进项目。**
---
## 2. 当前工作方式总原则
当前项目采用以下基本工作方式： - **唯一工作区：** `jianglab` - 
**主控制台：** Codex app - **辅助终端：** FinalShell - 
**正式代码与文档事实源：** `jianglab` 上的 Git 仓库 - 
**正式推送方式：** `jianglab -> GitHub` SSH 推送 当前阶段，不维护 
Windows 本地的第二份主副本。 所有正式开发、文档更新、Git 操作，都以 
`jianglab` 上的仓库为准。 ---
## 3. 当前阶段目标
当前项目已经从早期 OpenClaw 原生入口实验，转入：
> **AI 销售助手 V1 的定义、收敛与标准化 agent 开发阶段**
当前阶段最重要的目标不是立刻扩展大量功能，而是先建立： 1. 
清晰的项目主线 2. 稳定的文档体系 3. 可持续的 agent 开发工作流 4. 
小步、可 review 的任务推进方式 ---
## 4. 所有变化，先分成三类
以后每次有新想法、新需求或新问题时，先不要直接让 Codex 改代码。 
先判断它属于哪一类：
### A. 方向变化
指的是影响“项目现在到底要做什么”的变化，例如： - 主线变化 - V1 
范围变化 - 当前不做的内容变了 - 目标用户或目标场景变了 - 
产品从一个方向切到另一个方向
### B. 新任务
指的是在当前方向不变前提下，一个可以单独完成的小闭环，例如： - 
新建某个文档 - 增加一个数据模型 - 新做一个页面骨架 - 新增一个模块入口 
- 写一个 runbook
### C. 细节修订
指的是当前方向不变、当前功能也基本不变，只调整局部细节，例如： - 
按钮文案修改 - 标题修改 - 信息顺序调整 - 结果展示微调 - 
某个字段命名优化 ---
## 5. 三类变化分别怎么处理 5.1 方向变化：先改定义，再改实现
如果属于**方向变化**，不要先动代码。 正确顺序： 1. 更新 
`docs/00_project_overview.md` 2. 更新 `docs/01_prd/...` 3. 
必要时新增或更新 `docs/06_decisions/...` 4. 明确哪些旧任务失效 5. 
重新拆新的 task 6. 再让 Codex 开始执行
### 方向变化的判断标准
只要你发现自己在改这些内容，就说明已经不是“小修小补”： - 
“我们现在其实不是做这个产品” - “V1 先不做这个模块了” - 
“当前用户不是这类人” - “这个阶段应该先验证别的事情”
### 重要原则
> **方向变了，先改文档，不先改代码。**
---
## 5.2 新任务：先写 task，再开 Codex 线程
如果属于**新任务**，标准顺序是： 1. 明确目标 2. 写 task 文档到 
`docs/03_tasks/` 3. 再开一个 Codex 线程 4. 让 Codex 只做这个 task 5. 
做完后更新 task 状态与 handoff 6. review 后再进入下一个任务
### 一个合格 task 至少要说明
- 目标是什么 - 范围是什么 - 不做什么 - 涉及哪些文件 - 如何验收
### 重要原则
> **没有 task，就不要直接让 Codex 做正式开发任务。**
---
## 5.3 细节修订：作为 follow-up task 处理
如果功能已经基本完成，但还有细节要改，不要直接把它塞回旧任务里乱改。 
建议分情况处理：
### 小修订
例如： - 改一两句文案 - 改标题 - 调整说明顺序 可以作为一个小 
follow-up 处理。
### 中等修订
例如： - 改页面结构 - 改字段组织 - 改数据流 - 改多个文件 
建议新建一个新的 follow-up task，例如： - 
`task_005_home_copy_adjustments.md` - 
`task_006_product_profile_followup.md`
### 重要原则
> **功能完成后的细节修改，默认按新 task 或 follow-up task 管理。**
---
## 6. 日常标准工作流
以后日常推进项目，尽量按下面顺序进行：
### Step 1：先看当前项目状态
先确认： - 当前主线是什么 - 当前阶段是什么 - 当前最高优先级是什么 - 
是否已有正在进行中的 task 参考文件： - `docs/00_project_overview.md` 
- `docs/01_prd/...` - `docs/03_tasks/...` ---
### Step 2：判断当前问题属于哪一类
先问自己： - 这是方向变化？ - 这是一个新任务？ - 还是一个细节修订？ 
如果没分清，不要直接开始改。 ---
### Step 3：更新对应层级的文档
根据分类处理： - 方向变化 → 改 overview / PRD / decisions - 新任务 → 
新建 task - 细节修订 → 记到 follow-up task 或补充说明 ---
### Step 4：再让 Codex 执行
只有在任务边界清楚后，才开 Codex 线程。 推荐在任务提示里明确写： - 
先读 `AGENTS.md` - 先读当前 task - 不得超范围修改 - 做完后更新 task 
和 handoff ---
### Step 5：review
Codex 做完后，你至少检查： - 改了哪些文件 - 是否超出范围 - 
文档是否同步更新 - 是否提供了 handoff - 是否做了最基本的验证 
你不需要自己看懂全部代码，但一定要能看懂： - 这个任务做了什么 - 
有没有跑偏 - 下一步建议是什么 ---
### Step 6：收口
任务确认完成后： 1. 更新 task 状态为 `done` 2. 更新或新增 handoff 3. 
再决定下一个任务 4. 不在同一个线程里无限追加新需求 ---
## 7. 文档怎么配合工作流使用 7.1 `AGENTS.md`
用途： - 给 Codex 的仓库级规则文件 你怎么用： - 低频修改 - 
主要用于约束 agent 工作方式 - 不写产品细节 ---
## 7.2 `docs/00_project_overview.md`
用途： - 说明当前项目总状态 你怎么用： - 
当项目主线、当前阶段或当前工作方式变化时更新 - 不写任务细节 ---
## 7.3 `docs/01_prd/`
用途： - 写版本需求与参考研究 你怎么用： - 当 V1 
的边界、目标、流程逐渐明确时更新 - 产品设计收敛后重点维护这里 ---
## 7.4 `docs/02_specs/`
用途： - 技术方案说明 你怎么用： - 
当某个任务需要单独的技术设计说明时才写 - 不要求每个任务都先有 spec 
---
## 7.5 `docs/03_tasks/`
用途： - 管单个任务 你怎么用： - 正式开发任务都先写 task - 这是你和 
Codex 之间最重要的执行接口 ---
## 7.6 `docs/04_runbooks/`
用途： - 运行、构建、运维、工作流说明 你怎么用： - 查怎么做 - 
固化方法 - 减少以后重复讨论 ---
## 7.7 `docs/05_handoffs/`
用途： - 阶段性交接与任务结果记录 你怎么用： - 
每个重要任务完成后都要更新 - 用来告诉未来的你和 AI：现在到哪一步了 
---
## 7.8 `docs/06_decisions/`
用途： - 记录关键决策 你怎么用： - 只在真正重要的取舍时记录 - 
不写零碎决定 ---
## 8. 什么时候不应该直接让 Codex 开工
下面这些情况，不要一上来就让 Codex 改代码：
### 1. 你自己还没想清楚这是方向变化还是细节修订
先分类。
### 2. 你只是在“脑暴”，还没有任务边界
先写几行 task 或笔记。
### 3. 这个变化会影响版本范围
先改 PRD / overview。
### 4. 你只是想参考市面产品
先做参考研究，不要直接编码。
### 5. 当前线程已经完成原任务，你又临时想加三个新需求
应该新建 task，不要继续在旧线程追加。 ---
## 9. 一个任务完成后的最小收口标准
一个任务在被认为“完成”之前，至少应满足： 1. 任务目标已实现 2. 
没有明显超出范围 3. 相关文档已同步 4. task 状态已更新 5. handoff 已写 
6. 至少做了最小可行验证 如果这些没做完，不要轻易把任务当成真正完成。 
---
## 10. 当前阶段最推荐的推进顺序
在当前阶段，推荐按下面顺序推进： 1. 完成仓库规则文件 2. 完成总览文档 
3. 做定向参考研究 4. 写 V1 PRD 5. 再拆第一个正式产品 task 6. 用 Codex 
小步推进实现
### 当前不建议
- 一开始就并行很多功能线程 - 一开始就大改架构 - 
一开始就把细节全部定死 - 一开始就做自动触达 / CRM 扩展 ---
## 11. 典型工作场景示例 场景 A：产品需要改方向
例如： “我们先不做数字分身了，先做 AI 销售助手。” 处理方式： 1. 更新 
`00_project_overview.md` 2. 更新 PRD 3. 必要时写 decision 4. 暂停旧 
task 5. 重拆新 task 6. 再让 Codex 开工 ---
## 场景 B：要做一个新功能
例如： “先建立 ProductProfile 数据模型。” 处理方式： 1. 新建 task 2. 
明确范围和验收标准 3. 开 Codex 线程执行 4. 完成后更新 handoff ---
## 场景 C：功能做完了，但文案不满意
例如： “首页主按钮名字要改。” 处理方式： - 小修订：作为 follow-up 
task - 不要直接往旧任务里无限追加 ---
## 场景 D：还没想清楚细节，先想参考市面产品
处理方式： 1. 不急着让 Codex 改代码 2. 先写参考研究笔记 3. 
提炼设计原则 4. 再进入 PRD ---
## 12. 当前最重要的开发纪律
### 纪律 1
**方向变化先改文档，不先改代码。**
### 纪律 2
**新功能先有 task，再开线程。**
### 纪律 3
**细节修订默认按 follow-up task 管理。**
### 纪律 4
**一个线程尽量只做一个任务。**
### 纪律 5
**任务做完必须收口，不要无限续接。** ---
## 13. 一句话工作流总结
当前项目的标准推进方式是：
> **先判断是方向、任务还是细节，再更新对应层级文档，最后让 Codex 在明确边界内执行。**
