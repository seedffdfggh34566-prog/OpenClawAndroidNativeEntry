## 1. 决策目的

本文档用于正式记录当前项目在 **后端承载位置、手机端角色、OpenClaw 角色、未来迁移方向** 上的关键决策。

它回答的问题包括：

1. 当前为什么不采用手机端主存。
2. 当前为什么不直接把正式后端定义为公有云部署。
3. 当前为什么选择“本地服务器承载正式后端”。
4. OpenClaw 在系统中到底扮演什么角色。
5. 这项决策会约束哪些实现方式。
6. 哪些内容已经定下，哪些内容仍然保留后续选择空间。

本文档不是：

- 完整部署手册
- 完整数据库设计文档
- 云迁移实施方案
- 具体服务器采购说明

---

## 2. 决策背景

当前项目已经从早期 OpenClaw Android Native Entry / 宿主入口实验，切换到：

> **AI 销售助手 App V1**

当前主线已经明确为：

1. 产品学习
2. 获客分析
3. 结构化输出 / 报告

当前明确不优先推进：

- 联系方式抓取
- 自动触达
- 完整 CRM

同时，当前已明确的系统使用方向为：

- 手机安卓端继续保留价值，但更适合作为**控制入口**
- 权威主存不应落在手机端
- 当前阶段需要一个可持续、可运维、可逐步接入 agent 的正式后端承载点
- 后续存在多人使用、数据同步和未来迁云的可能性

在这一背景下，需要正式确定当前阶段的后端部署基线。

---

## 3. 最终决策

当前项目第一阶段正式采用以下路线：

> **本地服务器承载正式后端 + 手机端仅作控制入口 + 架构按未来可迁云设计**

这句话包含 4 个子决策：

### 3.1 决策 A：手机端不承担权威主存

手机端当前不作为：

- 权威业务主数据库
- 团队协作裁决点
- 正式结果对象的唯一真相层

手机端当前主要承担：

- 产品入口
- 控制入口
- 任务发起入口
- 结果查看入口
- 轻量编辑与确认入口

### 3.2 决策 B：第一阶段正式后端部署在本地服务器 / 电脑

当前正式后端第一阶段承载于本地服务器 / 开发主机环境，而不是手机端，也不是必须立刻上公有云。

当前后端承载内容包括：

- 业务对象主存
- 分析任务编排
- 结果沉淀
- 轻量历史记录
- 与 OpenClaw runtime 的协同

### 3.3 决策 C：OpenClaw 作为 agent runtime / execution layer

OpenClaw 当前扮演：

- agent runtime
- 模型调用层
- 工具调用层
- 执行与编排层

OpenClaw 当前不承担：

- 产品业务主数据库
- 产品对象模型定义中心
- 最终产品层协作与权限中心

### 3.4 决策 D：架构按未来可迁云设计

虽然当前部署基线选定为本地服务器承载正式后端，但对象模型、服务边界、agent contract 和数据读写方式应按未来迁移到云端的可能性设计。

---

## 4. 为什么不采用手机端主存

### 4.1 原因一：当前产品已不再是单机宿主实验

当前项目已从早期宿主入口实验，转向 AI 销售助手 V1。  
V1 要沉淀的是：

- `ProductProfile`
- `LeadAnalysisResult`
- `AnalysisReport`
- `AgentRun`

这些都是正式业务对象，不应只存在于手机端本地状态或单次聊天上下文中。

### 4.2 原因二：手机端定位已经明确为控制入口

手机端继续非常重要，但它最适合承担的是：

- 发起流程
- 查看状态
- 查看结果
- 轻量编辑

而不是承担正式主存、任务编排和协作主真相。

### 4.3 原因三：后续多人使用与同步需求不适合手机主存

如果后续存在：

- 多人使用
- 多设备访问
- 历史结果复看
- 统一权限
- 统一结果沉淀

则手机端主存会迅速带来：

- 数据孤岛
- 同步复杂度上升
- 结果一致性差
- 后续扩展困难

### 4.4 原因四：手机端主存会让 agent 执行与正式结果边界混乱

如果把主存落在手机端，后续会出现：

- agent 在哪执行
- 结果写回哪里
- 不同设备如何同步
- 任务运行记录如何统一保留

这些问题会把产品层和执行层重新混在一起。

---

## 5. 为什么当前不直接定为“公有云正式后端”

### 5.1 当前阶段重点仍是 V1 闭环收口，而不是正式 SaaS 交付

当前阶段的重点仍然是：

- 把对象模型做对
- 把主流程做对
- 把产品学习、获客分析、报告输出串成稳定闭环
- 建立可持续的 agent 开发工作流

如果现在立即把重心切到公有云正式交付，会过早引入：

- 云资源与网络运维
- 安全组 / 域名 / 公网入口
- 托管数据库与对象存储
- 多租户、账号、部署流程

这些问题当前都不是最优先的产品阻塞点。

### 5.2 当前已有稳定可用的本地开发与运维主机条件

当前 `jianglab` 已经具备：

- 唯一正式工作区
- 稳定远程运维能力
- 持续开发环境
- 后续本地模型与 agent 节点可能性

因此，它天然适合作为第一阶段正式后端承载点。

### 5.3 当前更需要的是“服务化边界”，不是“云化表象”

当前真正必须先做对的是：

- 对象模型
- API 边界
- agent runtime 与产品层分层
- 数据主真相位置

这些事情无论后端跑在本地服务器还是云端，都必须先做对。  
当前先选本地服务器承载正式后端，更利于控制复杂度。

---

## 6. 为什么当前采用“本地服务器承载正式后端”

### 6.1 能兼顾控制感、迭代速度与工程连续性

本地服务器路线的优势包括：

- 使用已有主机
- 开发、调试、运维路径更短
- 与当前 `jianglab` 工作流自然衔接
- 便于先把 V1 跑通，而不是先做大量部署工作

### 6.2 有利于保持系统分层清晰

采用本地服务器承载正式后端后，当前系统边界更容易明确为：

- 手机端：控制入口
- 本地服务器：正式后端
- OpenClaw：runtime / execution layer
- 产品系统：正式业务对象与状态沉淀层

### 6.3 有利于未来把后端迁到云端，而不是把手机端硬改成 SaaS

如果第一阶段后端就已经是正式后端，那么未来迁云时，本质上只是：

- 更换部署位置
- 保持对象模型
- 保持 API 边界
- 保持产品流程

这比“先做手机本地主存，再改成云后端”要健康得多。

---

## 7. 这项决策带来的明确约束

以下约束从当前开始视为有效。

### 7.1 不允许手机端成为正式业务主真相

不允许把以下内容长期只存在手机端：

- 正式 ProductProfile
- 正式 LeadAnalysisResult
- 正式 AnalysisReport
- 正式 AgentRun

### 7.2 不允许 OpenClaw 直接等于产品本体

不允许把以下职责直接塞给 OpenClaw：

- 产品数据库
- 产品对象模型中心
- 结果历史唯一存储
- 协作和状态中心

### 7.3 不允许本地部署方式和产品逻辑强耦合

不允许形成这些实现：

- 只有某台机器上的本地 markdown / json 才是事实源
- 产品流程只能依赖某个本机路径或脚本手工串起来
- 手机端或单机状态成为唯一可复现结果的来源

### 7.4 所有正式业务对象默认以后端为准

以下对象默认定义为后端权威对象：

- `ProductProfile`
- `SourceArtifact`
- `Lead`
- `LeadResearchSnapshot`
- `LeadAnalysisResult`
- `AnalysisReport`
- `AgentRun`

---

## 8. 当前保留不定的内容

下面这些内容当前故意不定死。

### 8.1 不定死最终是否进入公有云

当前只是明确：

- 第一阶段先采用本地服务器承载正式后端
- 未来保留迁云能力

当前不承诺：
- 一定上公有云
- 一定做公有云 SaaS 首发

### 8.2 不定死数据库与对象存储具体产品

当前不强行定：

- PostgreSQL / MySQL / SQLite 的最终组合
- 对象存储用什么实现
- 向量检索最终选型

### 8.3 不定死最终是否支持专属云 / 私有化部署

当前只要求：

- 架构不要排斥未来迁移
- 对象模型和 API 边界保持独立

### 8.4 不定死实时协作和复杂权限系统

当前只预留后续扩展空间，不在第一阶段把：

- 多租户
- 复杂权限树
- 实时多方协作编辑

拉入 V1 基线。

---

## 9. 对后续开发的直接影响

### 9.1 对 PRD 的影响

PRD 中应正式体现：

- 手机端仅作控制入口
- 正式后端在本地服务器
- OpenClaw 作为 runtime
- 正式对象以后端为准

### 9.2 对 specs 的影响

后续 technical spec 应按以下分层写：

- 手机端
- 产品系统后端
- OpenClaw runtime
- 数据层

### 9.3 对 task 拆分的影响

后续 task 应避免直接从“手机端页面”视角拆分全部逻辑。  
至少要把以下内容拆清：

- 对象模型 task
- 产品学习流程 task
- 分析结果对象与页面 task
- 报告对象与页面 task
- 后端接口与状态 task

### 9.4 对实现方式的影响

应优先采用：

- 服务化边界
- 后端权威对象
- 控制入口式移动端
- agent runtime 与产品层分离

不应优先采用：

- 本地脚本拼接式产品逻辑
- 手机端本地状态直接充当正式数据
- OpenClaw 直接充当产品数据库

---

## 10. 当前一句话基线

从当前版本开始，正式采用以下架构基线：

> **本地服务器承载正式后端，手机端仅作控制入口，OpenClaw 作为 agent runtime / execution layer，产品正式业务对象以后端为权威真相，并按未来可迁云设计。**

> **手机端不做主真相层，第一阶段以后端服务化为前提，由本地服务器承载正式后端；这是一条为了控制复杂度、保持分层清晰并为未来迁云留出空间的阶段性基线决策。**
