## 1. 文档目的

本文档用于定义当前阶段的 **AI 销售助手 V1**。

它回答以下问题：

1. V1 现在到底做什么，不做什么。
2. V1 要服务什么用户、解决什么核心问题。
3. V1 的系统分层与部署前提是什么。
4. V1 的主流程是什么。
5. V1 至少需要哪些页面、对象和功能。
6. 后续任务应该如何拆分。

本文档不是：

- 完整商业计划书
- 详细数据库 schema 文档
- 完整 CRM 平台需求总表
- 自动触达产品规划
- 云部署实施方案

本文档只服务当前最小闭环：

> **产品学习 → 获客分析 → 结构化输出 / 报告**

---

## 2. 背景与机会判断

当前 AI 销售赛道中，已有大量产品覆盖企业数据库、线索筛选、联系方式获取、序列触达、CRM 协同等能力，但普遍有两个问题：

1. 对中小团队和个人销售来说，前期上手门槛高。
2. 大部分产品默认用户已经知道如何表达自己的产品、客户与销售路径，缺少“帮助用户先想明白自己卖什么、卖给谁”的环节。

当前 V1 的机会不在于立刻做出另一个大而全销售平台，而在于把最前面的认知工作做掉：

- 帮用户通过对话快速澄清产品理解
- 基于这一理解，自动生成获客分析结果
- 将结果交付成一份用户愿意保存和继续用的结构化报告

因此，当前 V1 的差异化核心是：

> **AI 对话式产品学习 + 基于产品画像的获客分析**

---

## 3. 当前版本定位

### 3.1 一句话定位

AI 销售助手 V1 是一个面向中小企业老板、销售负责人和个人 BD 的轻量 AI 销售产品：用户通过手机端控制入口发起产品学习与获客分析，由正式后端与 agent runtime 生成结构化产品画像和分析报告。

### 3.2 当前版本目标

让用户在一次完整使用中完成以下闭环：

1. 讲清楚自己的产品
2. 被 AI 引导补全关键销售信息
3. 自动生成结构化 ProductProfile
4. 自动得到第一版获客分析结果
5. 输出一份可复看、可编辑、可迭代的分析报告

### 3.3 当前版本明确不做

V1 不做以下内容：

- 联系方式抓取
- 电话 / 企微 / 邮件自动触达
- 完整 CRM
- 团队协同审批流
- 大规模客户库管理
- 自动化工作流搭建器
- 企业级 BI / 仪表盘
- 正式公有云 SaaS 首版上线
- 手机端主存

## 4. V1 架构前提与系统分层

本节不是详细技术 spec，而是当前版本的产品级系统前提。

### 4.1 当前部署基线

当前项目第一阶段正式采用以下路线：

- **本地服务器承载正式后端**
- **手机端仅作控制入口**
- **OpenClaw 作为 agent runtime / execution layer**
- **架构按未来可迁云设计**

### 4.2 手机端的角色

手机端当前承担：

- 登录与进入产品
- 发起产品学习流程
- 查看当前任务状态
- 查看与轻量编辑 ProductProfile
- 查看分析结果与报告
- 发起重新分析 / 继续完善

手机端当前不承担：

- 权威主数据库
- 正式业务对象主存
- 主要 agent 执行节点
- 团队协作裁决点

### 4.3 本地服务器的角色

本地服务器 / 电脑承载第一阶段正式后端，负责：

- 业务对象主存
- 分析任务编排
- 报告沉淀
- 结果复看与历史记录
- 后续团队协作能力预留

### 4.4 OpenClaw 的角色

OpenClaw 在系统中扮演：

- agent runtime
- 模型调用层
- 工具调用层
- 执行与编排层

OpenClaw 当前不直接承担：

- 产品业务主数据库
- 产品对象模型定义中心
- 权限与协作中心

### 4.5 产品系统自身的角色

产品系统负责：

- `ProductProfile`
- `LeadAnalysisResult`
- `AnalysisReport`
- `AgentRun`
- 历史记录与状态
- 权威业务对象沉淀

也就是说：

> 用户看到的正式结果，应沉淀为产品系统对象，而不是停留在一次聊天过程中。

### 4.6 未来迁云原则

虽然第一阶段正式后端部署在本地服务器，但以下内容应按未来可迁云设计：

- 对象模型
- API 边界
- agent contract
- 运行状态与结果对象的关系

当前不允许形成以下耦合：

- 只有某台机器上的本地文件才是事实源
- 只有本机路径和脚本才能驱动产品工作
- 只有端侧私有状态才能复现业务结果

## 5. 目标用户

### 5.1 核心目标用户

#### A. 中小企业老板 / 创始人
特点：
- 对业务熟，但未必系统化梳理过销售定位
- 希望快速判断“我的产品到底该卖给谁”
- 不一定有成熟销售团队

#### B. 销售负责人 / 商务负责人
特点：
- 手头有产品，但需要更高质量的潜客方向和销售切入点
- 需要快速生成一版团队可讨论的分析材料

#### C. 个人销售 / BD / 业务拓展
特点：
- 需要尽快理解产品和目标客群
- 需要一个结构化起点，而不是自己从零写分析文档

### 5.2 当前不优先覆盖的用户

- 大型企业复杂销售组织
- 强 CRM 依赖团队
- 以自动外呼 / 自动营销为刚需的团队
- 需要大量权限管理与审计的企业客户

## 6. 用户核心问题

V1 要优先解决的不是“我怎么自动发消息”，而是下面这些更前置的问题：

1. 我的产品到底是什么，应该怎么被说清楚。
2. 哪类客户最可能买，为什么。
3. 除了我直觉中的客户，还有没有上下游或邻近场景机会。
4. 哪些方向值得优先尝试，哪些方向应该放后。
5. 我能不能快速得到一份像样的分析结果，而不是只得到一段聊天记录。
6. 我得到的结果能不能沉淀成可复看的正式对象，而不是一次性对话。

---

## 7. 核心价值主张

### 7.1 对用户的核心价值

- 不需要先准备很完整的资料，也能开始分析
- AI 会主动追问关键问题，而不是等用户自己组织好全部思路
- 输出不是零散建议，而是一份结构化分析结果
- 能帮助用户发现优先行业、客户类型、使用场景与潜在机会
- 结果会沉淀为正式对象，便于复看、修订和继续推进

### 7.2 与常见销售工具的区别

常见销售工具更偏向：
- 数据查询
- CRM 管理
- 联系方式获取
- 自动触达

当前 V1 更偏向：
- 产品理解
- 销售定位澄清
- 获客方向研究
- 报告化输出

## 8. 产品范围

### 8.1 V1 In Scope

V1 范围内必须覆盖：

1. **产品学习对话**
   - 用户输入产品信息
   - AI 主动追问关键补充问题
   - 形成结构化产品画像草稿

2. **产品画像确认**
   - 用户查看和修订 ProductProfile
   - 系统明确哪些信息已充分，哪些信息仍模糊

3. **获客分析生成**
   - 基于 ProductProfile 生成目标行业、客户类型、使用场景、上下游机会等分析
   - 给出优先级和理由

4. **结构化分析报告输出**
   - 将结果整理成页面级或文档级报告
   - 报告能复看、继续编辑和后续迭代

5. **历史结果与状态入口（轻量）**
   - 用户至少能找到最近一次或若干次分析结果
   - 用户知道当前分析流程处于哪个状态

### 8.2 V1 Out of Scope

当前不进入 V1：

- 联系人数据库导出
- 邮件 / 电话 / 短信 / 企微触达
- 自动序列
- 线索库管理
- 团队协作评论 / 审批
- 外部 CRM 双向同步
- 自动监控和推送订阅
- 完整云端 SaaS 交付形态
- 复杂离线同步机制

---

## 9. V1 的主流程

### 9.1 核心主路径

建议 V1 主路径如下：

1. 用户进入手机端首页
2. 点击“开始分析”
3. 进入产品学习对话页
4. AI 基于用户输入发起追问
5. 后端形成 ProductProfile 草稿
6. 用户确认 / 修订 ProductProfile
7. 后端生成获客分析结果
8. 系统生成 AnalysisReport
9. 用户查看、复制、继续编辑或重新分析

### 9.2 该路径的设计要求

- 主路径必须简单清楚，不要出现过多岔路
- 第一次使用应尽量在单次连续流程中完成
- 不要求用户理解内部 Agent 结构
- 用户始终知道自己当前在哪一步、下一步会得到什么
- 用户知道结果是已保存对象，而不是一次性会话内容

## 10. 信息架构与核心页面

当前建议 V1 至少包含以下页面：

### 10.1 首页 / 开始分析入口

作用：
- 告诉用户产品是干什么的
- 引导用户开始第一轮分析
- 展示已有分析记录入口（轻量）
- 展示当前分析状态入口（轻量）

首页重点不在“功能导航”，而在“立刻开始”。

建议包含：
- 产品一句话说明
- 主按钮：开始分析
- 次按钮：查看历史 / 查看示例
- 最近一次分析状态（如有）

### 10.2 产品学习对话页

作用：
- 承载用户与 AI 的产品学习过程
- 通过追问澄清关键信息
- 驱动后端更新 ProductProfile 草稿

建议呈现：
- 用户输入区
- AI 提问与总结区
- 当前已收集关键信息的简要摘要或阶段提示
- 当前处理状态提示
- “进入下一步”按钮（当信息达到最低完整度时出现）

### 10.3 产品画像确认页

作用：
- 把聊天结果收口成结构化 ProductProfile
- 让用户明确看到 AI 当前对产品的理解
- 确认后作为后续分析正式输入

建议包含：
- 产品摘要
- 核心卖点
- 目标客户类型
- 典型使用场景
- 行业属性
- 购买动因
- 常见顾虑
- 邻近行业 / 上下游机会线索
- 编辑与确认入口
- 当前版本说明

### 10.4 获客分析结果页

作用：
- 展示 AI 基于 ProductProfile 给出的第一版分析
- 让用户理解优先方向和理由

建议包含：
- 推荐优先行业
- 推荐优先客户类型
- 推荐优先场景
- 上下游 / 邻近行业机会
- 优先级排序
- 每条建议的理由
- 风险与限制说明
- 重新分析入口

### 10.5 分析报告页

作用：
- 将前述结果以更稳定、更适合复看的方式展示
- 承担正式输出对象的查看入口

建议包含：
- 产品理解摘要
- 目标客户画像摘要
- 获客方向建议
- 优先级排序
- 场景与行业机会说明
- 下一步建议动作
- 更新时间与版本信息

### 10.6 历史结果入口（轻量）

作用：
- 让用户找到最近生成的 ProductProfile / Report
- 避免结果只存在于单轮会话中

V1 不需要做成完整工作台，但至少应支持：
- 最近一次分析入口
- 最近若干条历史记录入口

## 11. 关键对象定义

本节为产品定义，不是正式数据库 schema。

### 11.1 ProductProfile

ProductProfile 是 V1 的核心中间对象。

它至少应包含：

- `id`
- 产品名称
- 产品一句话描述
- 产品类别
- 核心卖点
- 目标客户类型
- 主要使用场景
- 当前成交模式 / 交付方式
- 购买动因
- 常见顾虑 / 阻力
- 所在行业
- 可延伸行业 / 邻近场景
- 上下游关系线索
- 当前已知优势与短板
- `status`
- `version`
- `updated_at`

要求：
- 它不能只是聊天记录
- 它必须能被后续分析流程复用
- 用户应可对其进行确认和修订
- 它以后端版本为准

### 11.2 LeadAnalysisResult

LeadAnalysisResult 是基于 ProductProfile 生成的分析结果对象。

至少包含：
- `id`
- `product_profile_id`
- 推荐优先行业列表
- 推荐优先客户类型列表
- 典型使用场景列表
- 上下游 / 邻近行业机会
- 优先级排序
- 每项推荐的理由
- 风险与限制说明
- 下一步建议动作
- `status`
- `version`

要求：
- 不只是列清单，还要体现优先级与理由
- 应明确区分“现在优先做”和“后续可观察”的方向
- 结果对象应可复看、可重跑、可替换版本

### 11.3 AnalysisReport

AnalysisReport 是用户最终可阅读和复用的输出对象。

至少包含：
- `id`
- `product_profile_id`
- `lead_analysis_result_id`
- 产品理解摘要
- 目标客户画像摘要
- 获客方向建议
- 优先级排序
- 场景 / 行业机会说明
- 风险提醒
- 进一步验证建议
- `status`
- `version`

要求：
- 适合长期沉淀
- 不局限于单轮对话
- 后续应支持继续迭代更新

### 11.4 AgentRun

AgentRun 是系统内部执行过程的记录对象。

至少包含：
- `id`
- `run_type`
- `input_refs`
- `output_refs`
- `status`
- `started_at`
- `finished_at`
- `error_summary`

要求：
- 用于追踪一次产品学习 / 分析 / 报告生成过程
- 便于后续定位问题与解释结果来源

## 12. 功能需求

## 12.1 产品学习模块

### 目标
帮助用户通过对话完成第一版产品理解梳理。

### 核心要求
- 用户可用自然语言描述产品
- AI 能识别信息缺口并主动追问
- AI 追问应围绕产品、客户、场景、优势、购买动因等关键维度展开
- 多轮对话后，系统应能形成初版 ProductProfile
- ProductProfile 草稿应写入后端对象，而不是停留在会话里

### 最低可用标准
- 至少能从自由输入中提炼出 5~8 个关键字段
- 至少能补问 3~5 个关键问题
- 至少能生成一版用户可读的结构化摘要

## 12.2 产品画像确认模块

### 目标
让用户校正 AI 当前理解，避免后续分析建立在错误前提上。

### 核心要求
- ProductProfile 以结构化方式展示
- 用户可编辑或确认主要字段
- 系统可提示“信息较弱 / 需补充”的部分
- 确认后的对象可作为正式分析输入

### 最低可用标准
- 用户能看懂 AI 当前对产品的理解
- 用户能修改关键字段
- 修改后可重新进入分析

## 12.3 获客分析模块

### 目标
根据 ProductProfile 输出潜客方向与优先级建议。

### 核心要求
- 输出不只是推荐对象，还要有推荐理由
- 至少覆盖行业、客户类型、场景、上下游 / 邻近机会
- 必须体现优先级排序
- 需给出部分风险和限制说明
- 分析结果应沉淀为正式对象

### 最低可用标准
- 至少输出 3 类潜在客户方向
- 至少给出每类方向的理由
- 至少明确一个优先顺序

## 12.4 报告输出模块

### 目标
将结果从聊天记录沉淀为可复看的结构化输出。

### 核心要求
- 输出适合阅读
- 结构清晰
- 能作为后续销售讨论或继续优化的基础材料
- 支持后续重跑与版本迭代

### 最低可用标准
- 用户可在单独页面查看完整报告
- 报告内容不是简单拼接聊天记录
- 报告至少具备标题、分节、小结与下一步建议

## 12.5 历史结果与状态模块（轻量）

### 目标
避免用户结果只存在于单轮会话中。

### 核心要求
- 用户至少能找到最近一次或若干次历史结果
- 用户能知道当前分析流程状态
- 用户能从历史入口进入已有 ProductProfile / Report

### 最低可用标准
- 至少有最近一次结果入口
- 至少能看到当前一次任务的状态

## 13. 非功能要求

### 13.1 易用性
- 首次主路径应清晰
- 不要求用户预先理解复杂术语
- 交互应以简单引导为主，而不是复杂配置

### 13.2 速度感
- 第一版结果应尽快出来
- 用户不能长时间处在“不知道系统在做什么”的状态

### 13.3 可理解性
- 系统输出应带有理由
- 用户应理解为什么某类行业 / 客户被优先推荐

### 13.4 可迭代性
- ProductProfile、LeadAnalysisResult、AnalysisReport 应天然支持后续更新
- 后续可在不重做整体架构的前提下增强数据层与评分逻辑

### 13.5 系统分层清晰
- 手机端不是权威主存
- 正式对象以后端为准
- OpenClaw 是 runtime，不是产品本体
- 产品层对象与执行层应区分

### 13.6 可迁移性
- 第一阶段部署在本地服务器
- 未来迁云应尽量不改对象模型和页面逻辑
- 当前实现不应把关键流程写死在本机耦合逻辑中

---

## 14. 明确不做什么

为避免需求膨胀，当前版本明确不做：

1. 联系方式获取
2. 企微 / 电话 / 邮件自动触达
3. CRM 客户管理体系
4. 大规模线索库导入导出
5. 复杂权限体系
6. 团队协同评论流
7. 自动化工作流编排器
8. 企业级数据看板
9. 完整云端 SaaS 首版部署
10. 手机端主存
11. 本地部署与产品逻辑强耦合

---

## 15. 版本成功标准

### 15.1 产品层成功标准

当一个新用户第一次使用时，至少应能做到：

1. 能顺利完成一次产品描述
2. 能被 AI 追问到关键信息补全
3. 能看到一版结构化 ProductProfile
4. 能得到一版像样的获客分析结果
5. 能看到一份可复看的分析报告
6. 能在后续重新找到这份结果

### 15.2 体验层成功标准

- 用户知道每一步自己正在做什么
- 用户不会误以为这是 CRM 或触达工具
- 用户在第一次使用后，能明确感受到“我对该卖给谁更清楚了”
- 用户知道结果已经被保存，而不是只能回看聊天内容

## 16. 验收标准（面向后续 task）

下面这些标准可直接作为后续任务验收参考。

### 16.1 产品学习任务验收
- 能进入产品学习流程
- AI 至少发起若干关键追问
- 对话结束后能生成初版 ProductProfile
- ProductProfile 草稿对象被正式写入后端

### 16.2 产品画像任务验收
- ProductProfile 以结构化方式呈现
- 用户能编辑关键字段
- 编辑结果可用于后续分析

### 16.3 获客分析任务验收
- 能基于 ProductProfile 生成一版分析结果
- 结果至少包含方向、优先级、理由
- 结果不只是原始文本堆叠
- 结果对象被正式保存

### 16.4 报告输出任务验收
- 能生成单独报告页
- 报告有清晰分节
- 报告适合复看和继续迭代
- 报告对象可从历史入口再次打开

### 16.5 首页 / 历史入口任务验收
- 用户能从首页开始第一轮分析
- 用户能找到最近一次分析或报告入口
- 当前分析状态可见

## 17. 风险与注意事项

### 17.1 风险一：被成熟竞品带偏，范围迅速膨胀
应对：
- 持续坚持 V1 只做前两步加报告输出
- 不把联系方式、触达、CRM 拉入首版

### 17.2 风险二：产品学习阶段做成普通聊天机器人
应对：
- 强制要求 ProductProfile 作为结构化对象产出
- 不能只停留在对话表面

### 17.3 风险三：分析结果不够结构化，只是大段文字
应对：
- 在定义层先把 LeadAnalysisResult 和 AnalysisReport 结构写清楚
- 页面与输出都围绕结构组织

### 17.4 风险四：首页与 onboarding 过于概念化
应对：
- 首页围绕“开始分析”而不是“平台能力展示”设计
- 第一次使用快速进入产品学习流程

### 17.5 风险五：当前本地服务器部署导致后续迁云困难
应对：
- 明确产品层对象与 API 边界
- 明确 OpenClaw runtime 与业务主存分层
- 不做本地耦合型实现
