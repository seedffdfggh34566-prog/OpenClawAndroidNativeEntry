## 1. 文档目的

本文档用于承接 V1 closeout 之后的下一阶段规划。

V1 已经证明：

```text
产品学习 -> 产品画像确认 -> LeadAnalysis LLM -> 结构化报告 -> Android 真机复看
```

可以跑通，并且具备 demo-ready release candidate / learning milestone 价值。

V2 要回答的问题不是“继续 polish V1”，而是：

1. 下一阶段产品差异化是否成立。
2. 系统是否从“给方向建议”升级到“轻量线索研究”。
3. 后端是否允许主动联网 / 搜索。
4. 产品学习是否从表单式录入升级为 AI 引导式对话。
5. 数据层是否从 V1 的 4 个对象升级为可追踪、可验证、可复用的研究对象体系。

本文档不是：

- 已冻结的开发任务
- 完整商业计划书
- 正式数据库 schema
- 合规政策文件
- CRM / 自动触达 / 联系人抓取系统 PRD

---

## 2. 当前结论

V2 暂定方向为：

> **面向中小企业老板 / 销售负责人的 AI 销售助手，从 V1 的“获客方向分析”升级为“带来源证据的轻量线索研究”。**

核心变化：

1. 用户仍从“我卖什么”开始。
2. AI 不再只输出抽象行业和客户类型，而是可以主动联网搜索，给出一批具体公司 / 机构候选。
3. 每个候选必须有来源、匹配理由、不确定性和下一步验证动作。
4. 产品学习入口从表单式录入逐步转为 AI 引导式对话。
5. 后端数据层必须显式保存对话、产品画像版本、搜索来源、候选公司、联系方式和研究结果之间的关系。

当前仍未冻结：

- V2 是否直接作为 MVP。
- V2 是否需要正式云部署。
- V2 是否需要账号、多用户、租户隔离和权限。
- V2 是否需要付费前成本治理。
- V2 是否继续使用 Tencent TokenHub `minimax-m2.5` 作为默认 LLM。
- 个人联系方式的产品呈现边界、合规文案和保留策略。

## 3. 目标用户

### 3.1 核心用户

V2 优先服务：

- 中小企业老板
- 创始人
- 销售负责人
- 商务负责人

这些用户通常具备以下特点：

- 对自己的业务熟，但不一定有系统化获客方法。
- 想知道“我应该先找哪些具体客户”。
- 没有成熟的数据团队或销售运营团队。
- 可以接受 AI 辅助研究，但需要结果可解释、可核验。
- 更关心第一批可验证线索，而不是完整 CRM。

### 3.2 暂不优先用户

V2 暂不优先覆盖：

- 大型企业复杂销售组织
- 强 CRM 工作流团队
- 以大规模联系人采集为核心诉求的团队
- 以自动外呼、自动邮件、自动企微触达为核心诉求的团队
- 需要严格法务、审计、权限和数据保留策略的企业客户

## 4. V2 核心问题

V1 能回答：

- 我的产品适合哪些行业。
- 哪些客户类型可能优先。
- 有哪些场景、风险和下一步建议。

V2 要进一步回答：

- 这些方向下，第一批可以研究的具体公司是谁。
- 为什么这些公司可能匹配。
- 这个判断来自哪些公开来源。
- 还有哪些关键事实没有验证。
- 用户下一步应该手动验证什么、怎么开口、优先联系谁。

也就是说，V2 的重点不是“替用户自动销售”，而是：

> **把销售前期从抽象方向推进到可核验的第一批候选线索。**

---

## 5. V2 产品范围

### 5.1 In Scope

V2 规划范围内包含：

1. **AI 引导式产品学习**
   - Android 端不再依赖纯表单作为唯一入口。
   - AI 通过对话追问产品、客户、场景、地区、客单价、交付方式、成交约束。
   - 对话内容不直接等于业务真相，后端仍沉淀为 `ProductProfile`。

2. **产品画像版本化**
   - 用户每轮补充都能形成可追踪的产品画像变化。
   - 后续研究必须引用某个明确的 ProductProfile 版本。

3. **轻量线索研究**
   - 后端允许主动联网 / 搜索。
   - 搜索以中文公开网页为优先来源。
   - 系统可以产出具体公司 / 机构候选。
   - 每个候选必须包含来源证据和匹配理由。

4. **来源证据保存**
   - 搜索结果、网页来源、摘录摘要、检索时间必须可追踪。
   - 研究结果不能只保存 LLM 的最终结论。

5. **联系方式的受控展示**
   - 允许保存和展示公开来源中的公司联系方式。
   - 个人联系方式只允许在明确公开来源、人工核验提示和用途限制下出现。
   - 不做批量采集、批量导出、自动触达。

6. **销售行动建议**
   - 对每个候选给出下一步验证动作。
   - 对优先候选给出人工跟进切入点。
   - 输出仍应沉淀为结构化对象，而不是只存在于聊天记录。

### 5.2 Out of Scope

V2 暂不进入：

- Web 前端
- 完整 CRM
- 自动邮件 / 短信 / 企微 / 电话触达
- 自动外呼
- 批量联系人抓取
- 批量导出联系人库
- 购买第三方客户数据库
- 大规模爬虫系统
- 团队协作审批流
- 企业权限和审计体系
- 完整商业化计费系统

## 6. 交互形态

### 6.1 不做 Web 前端

V2 暂不做 Web 前端。

当前前端方向仍为：

- Android 作为主控制入口。
- 后端作为正式业务系统。
- 开发者调试入口可以继续保留 dev-only 工具。

这意味着 V2 不应把“ChatGPT-like 前端”理解为 Web 版聊天产品，而应理解为：

> **Android 内的对话式产品学习入口 + 结构化卡片和结果页。**

### 6.2 AI 引导式产品学习

V2 目标不是复制通用 LLM 聊天框，而是把聊天收束到销售业务对象。

建议交互结构：

```text
用户自然语言描述产品
    ↓
AI 追问关键销售信息
    ↓
右侧 / 下方展示当前 ProductProfile 摘要
    ↓
用户补充或纠正
    ↓
后端生成 ProductProfileRevision
    ↓
达到最低完整度后进入线索研究
```

关键原则：

- 对话是输入通道，不是最终事实。
- ProductProfile 是正式业务真相。
- 用户必须能看到 AI 当前理解。
- 用户必须能纠正关键字段。
- 后续搜索必须基于已确认或足够明确的产品画像版本。

---

## 7. 联网 / 搜索原则

V2 允许系统主动联网 / 搜索，但默认遵循以下约束：

1. 搜索目标优先为中文公开网页。
2. 每个候选公司必须至少有一个可追踪公开来源。
3. LLM 可以总结来源，但不能伪造来源。
4. 不确定信息必须显式标记。
5. 搜索结果必须与 ProductProfile 版本绑定。
6. 搜索过程应记录 query、source、retrieved_at 和使用结果。
7. 搜索失败时应给出失败原因，而不是编造候选。

可接受来源示例：

- 企业官网
- 政府 / 园区 / 协会公开页面
- 招投标 / 采购公告
- 新闻稿
- 行业媒体文章
- 招聘信息
- 公开黄页或平台页

需要谨慎的来源：

- 内容农场
- 无法判断更新时间的聚合页
- 转载链条不清晰的文章
- 只有片段、无稳定 URL 的搜索摘要
- 个人联系方式聚合站

## 8. 联系方式原则

V2 可以处理联系方式，但必须把“公司联系方式”和“个人联系方式”分开。

### 8.1 公司联系方式

允许范围：

- 公司公开电话
- 公司公开邮箱
- 官网联系页
- 公司公开地址
- 公开客服入口

要求：

- 必须记录来源。
- 必须记录 retrieved_at。
- 必须标记为人工验证建议。
- 不得自动触达。

### 8.2 个人联系方式

个人联系方式属于高风险信息。

V2 若展示个人联系方式，必须满足：

- 来源为公开页面。
- 明确记录来源 URL。
- 明确标记 `is_personal = true`。
- 明确标记 `manual_verify_required`。
- 不做批量抓取。
- 不做批量导出。
- 不做自动触达。

默认产品策略应偏保守：

> **V2 可以让用户知道“公开页面上存在某个联系入口”，但不把个人联系方式采集变成核心卖点。**

## 9. 核心对象

V2 应复用 V1 的核心对象：

- `ProductProfile`
- `LeadAnalysisResult`
- `AnalysisReport`
- `AgentRun`

同时新增更细的数据对象，支持对话、版本、搜索、来源和候选线索的可追踪关系。

建议新增对象：

- `ProductLearningSession`
- `ConversationMessage`
- `ProductProfileRevision`
- `LeadResearchResult`
- `ResearchSource`
- `ResearchQuery`
- `SearchResult`
- `CompanyCandidate`
- `ContactPoint`
- `SalesActionCard`

详细数据模型见：

- `docs/architecture/data/v2-lead-research-data-model.md`

## 10. V2 主流程

建议主路径：

```text
开始分析
    ↓
AI 引导式产品学习
    ↓
ProductProfile 草稿 / 修订
    ↓
用户确认或达到研究门槛
    ↓
LeadAnalysis 生成研究方向
    ↓
LeadResearch 主动联网搜索
    ↓
候选公司 / 来源 / 联系入口 / 不确定性沉淀
    ↓
生成销售行动建议
    ↓
报告化输出与历史复看
```

与 V1 的差异：

- V1 到 `LeadAnalysisResult` 后主要输出方向和报告。
- V2 在 `LeadAnalysisResult` 之后新增 `LeadResearchResult`，把方向落到具体候选和证据。

## 11. 结果形态

V2 的用户可见结果不应只是报告，而应至少包含三类输出：

1. **研究摘要**
   - 本轮基于哪个产品画像版本。
   - 搜索了哪些方向。
   - 得到多少候选。
   - 哪些结论确定，哪些待验证。

2. **候选公司卡片**
   - 公司名称
   - 匹配理由
   - 来源证据
   - 置信度
   - 不确定性
   - 可用联系入口
   - 下一步人工验证动作

3. **销售行动建议**
   - 优先跟进顺序
   - 第一轮开场切入点
   - 建议核验的问题
   - 不建议优先投入的方向

## 12. 成功标准草案

当前用户尚未厘清成功标准，因此本节只作为候选草案，不作为冻结验收口径。

### 12.1 最小产品成功标准

一轮 V2 成功应满足：

- 用户能通过对话完成 ProductProfile，而不是必须填完整表单。
- 系统能基于 ProductProfile 产出 10 到 20 个具体候选公司 / 机构。
- 每个候选至少有 1 个可追踪公开来源。
- 每个候选有清晰匹配理由。
- 每个候选有不确定性或待验证点。
- 每个候选有下一步人工验证动作。
- 报告和候选结果可复看。

### 12.2 质量标准候选

建议后续用真实中文样例评估：

- 8/10 样例中，用户认为候选方向“有参考价值”。
- 0 个无来源编造公司。
- 0 个无来源编造联系方式。
- 80% 以上候选能从来源中解释匹配理由。
- 研究结果中明确区分“已证据支持”和“推测”。
- 平均每轮成本和耗时可被开发者观察。

### 12.3 MVP 判断标准候选

如果 V2 要进入 MVP，至少还需要补齐：

- 正式部署形态。
- 账号 / 数据隔离。
- 搜索与 LLM 成本上限。
- trace / logging / 错误排查。
- 联系方式展示边界。
- 用户数据保留策略。
- 基础安全和密钥管理。

## 13. 技术和产品边界

### 13.1 后端仍是主真相

V2 继续沿用 V1 的基本架构原则：

- Android 是控制入口。
- 后端是正式业务系统。
- Runtime 负责执行和 draft payload。
- 正式对象写回由 backend services 裁决。
- 聊天、搜索、LLM 输出都不能直接替代正式对象。

### 13.2 Agent 化方向

V2 可以比 V1 更 agent 化，但应避免把 agent 化等同于“自由运行”。

建议拆分：

- ProductLearning Agent：负责追问和结构化产品画像草稿。
- LeadAnalysis Agent：负责生成研究方向和假设。
- LeadResearch Agent：负责搜索、来源筛选、候选生成。
- ReportGeneration Agent：负责面向用户的研究报告表达。

每个 Agent 必须：

- 有明确输入对象。
- 有明确输出 draft schema。
- 由后端裁决写回。
- 通过 `AgentRun` 或后续扩展运行记录追踪。

## 14. 主要风险

### 14.1 搜索结果质量风险

公开网页可能过时、重复、广告化或信息不完整。

缓解方向：

- 保存来源。
- 标记来源类型和可信度。
- 输出不确定性。
- 禁止无来源候选进入正式结果。

### 14.2 个人信息风险

联系方式处理容易让产品滑向联系人抓取。

缓解方向：

- 公司联系入口优先。
- 个人联系方式默认保守展示。
- 不做批量抓取、批量导出、自动触达。
- 明确人工验证提示。

### 14.3 成本和延迟风险

联网搜索 + 多轮 LLM 会显著高于 V1 成本。

缓解方向：

- 先做小批量研究。
- 每轮限定 query 和候选数量。
- 记录 usage、latency、source_count。
- 后续再进入缓存、限流和预算控制。

### 14.4 产品差异化风险

如果 V2 只是“把搜索结果喂给 LLM 生成报告”，仍可能不比通用 LLM 客户端强。

缓解方向：

- 强制结构化对象沉淀。
- 强制来源绑定。
- 强制候选卡片和下一步动作。
- 支持产品画像版本化和研究历史复用。

---

## 15. 待决问题

进入实现前至少需要明确：

1. V2 是继续 prototype，还是目标为 MVP。
2. 是否必须云部署。
3. 是否需要账号和数据隔离。
4. 搜索 provider 使用什么能力。
5. 是否允许长期保存网页摘录。
6. 个人联系方式展示到什么粒度。
7. 每轮最多生成多少候选。
8. 每轮最多执行多少搜索 query。
9. 是否将 ReportGeneration 升级为 LLM。
10. 是否需要成本上限、超时兜底和 fallback。
11. V2 成功标准最终采用哪些指标。

---

## 16. 推荐下一步

推荐顺序：

1. 冻结 V2 PRD v0.2：明确是否以 MVP 为目标。
2. 冻结 V2 data model v0.2：明确哪些对象进入首批实现。
3. 创建 V2 task queue：先做后端数据模型和 contract，不先做大 UI。
4. 做 LeadResearch 最小后端 spike：固定输入 ProductProfile，返回带来源的候选。
5. 做真实中文样例 eval：验证候选质量、来源质量、成本和耗时。

当前不建议：

- 直接开工 Android 大改。
- 直接做 Web。
- 直接做联系人抓取。
- 直接把 V2 包装成 MVP。
- 在未明确搜索 provider 和数据保留策略前做生产化。
