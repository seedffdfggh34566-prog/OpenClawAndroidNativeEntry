## 4. V1 为什么不是 MVP

V1 不进入 MVP 的原因：

1. **没有生产部署形态**
   - 当前仍以 jianglab 本地 backend、SQLite、adb reverse 和本地真机 demo 为主。
   - 未定义正式云部署、环境隔离、数据保留和运维方式。

2. **没有多用户、权限或数据隔离**
   - 当前对象模型可支持后续扩展，但 V1 没有用户账号、租户隔离、团队权限或审计能力。

3. **没有 LLM fallback**
   - ProductLearning / LeadAnalysis 失败时当前按 AgentRun failed 处理。
   - 还没有 heuristic fallback、重试策略或供应商降级策略。

4. **ReportGeneration 仍是 heuristic**
   - 报告结构已经像交付物，但最终报告没有独立 LLM 生成能力。
   - 当前报告质量上限依赖 LeadAnalysis 结构化结果和后处理。

5. **Inspector 是 dev-only**
   - LLM Run Inspector 适合开发者本地观察 raw model output。
   - 它没有鉴权、多用户隔离、长期存储、安全审计或生产可观测方案。

6. **成本和延迟只做到观测，未做到治理**
   - token usage 已可见。
   - TTFT 和 run duration 已有证据。
   - 但 V1 没有预算控制、限流、成本看板、超时兜底或 SLA。

7. **产品表达仍是 demo 闭环**
   - 当前足够演示“用户输入产品 -> 生成获客分析报告”。
   - 但还不能承诺正式客户可长期使用、多人协作、数据管理或稳定交付。

## 5. V1 停止条件

V1 收口后，默认不再继续执行以下任务：

- 不继续做 V1 UI polish。
- 不继续扩大 V1 demo 样例库。
- 不继续调 V1 prompt。
- 不把 V1 改成 CRM。
- 不加入联系人抓取、自动触达、企微、邮件、外呼或导出。
- 不把 V1 当前本地 demo 形态包装成 MVP。

允许的 V1 回访范围：

- 修复阻断 demo 复现的严重 bug。
- 补充事实性证据或修正文档错误。
- 作为下一阶段规划的输入材料。
